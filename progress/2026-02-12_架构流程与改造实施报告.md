# 文献分析平台架构梳理与改造实施报告（2026-02-12）

## 1. 当前项目实际架构（基于代码实测）

### 1.1 前端架构（React + AntD + SSE）
- 入口页面：`frontend/src/pages/Chat/index.tsx`
- 数据服务：`frontend/src/services/agents.ts`、`frontend/src/services/rag.ts`
- 展示组件：`frontend/src/components/ChatChartRenderer/index.tsx`
- 核心机制：
  - 输入问题后通过 `POST /api/v1/agent/stream` 建立 SSE 流。
  - 前端按 `routing/status/chunk/references/metadata/done` 事件逐步渲染。
  - 对话历史通过 `/rag/conversations` 拉取并在左侧展示。

### 1.2 后端架构（FastAPI + RAG + Multi-Agent + Skills）
- 应用入口：`backend/main.py`
- API 路由层：`backend/app/api/v1/*.py`
  - `rag.py`：RAG 问答与历史对话
  - `agents.py`：多智能体路由与流式输出
- Agent 协调层：`backend/app/agents/coordinator.py`
  - 路由 4 类 Agent：Retriever / Writer / Analyzer / Search
- RAG 引擎层：`backend/app/rag/engine.py`
  - Embedding: BGE-M3
  - 检索：Milvus 向量检索 +（可选）ES 混合检索
  - 生成：LLM（OpenAI 兼容接口）
- Skills 层：`backend/app/skills/*`
  - academic / analysis / utility / visualization 四类技能

### 1.3 数据存储与流通
- PostgreSQL：用户、项目、文献主数据、`conversations`（历史消息 JSON）
- MongoDB：文献分块文本
- Milvus：分块向量索引
- Redis：缓存与部分状态
- 核心数据流：
  1. 用户消息 -> Agent 路由
  2. Retriever 场景 -> RAG 检索（Milvus/ES + MongoDB）
  3. LLM 生成 -> SSE 分块返回
  4. references/metadata + answer 持久化到 `conversations.messages`
  5. 历史回放时从会话 JSON 恢复上下文与引用

## 2. 四个问题的根因分析

### 问题1：历史对话查看时引用来源为空
根因：
- 历史接口只返回 `role/content/created_at`，丢失 `references/metadata/agent_type`。
- `agent/stream` 保存历史时 assistant 消息未持久化 references。
- 前端切换历史会话时主动 `setReferences([])`，未从历史消息恢复引用。

### 问题2：智能体答案质量与展示样式需优化
根因：
- Prompt 对输出结构约束偏弱，容易出现大段无层次文本。
- 非 Retriever Agent 输出缺乏统一格式化。
- 引用列表缺乏可读性字段（如相关度、标题补全）。

### 问题3：skills 是否有缺陷、是否要补强
发现：
- `ChatOpenAI` 在部分 skill 中配置参数不统一（老写法与新写法混用）。
- 模型配置字段存在 `LLM_*` 与 `OPENROUTER_*` 分散引用风险。
- `utility_skills.py` 与 `util_skills.py` 职责可工作，但命名容易混淆。

### 问题4：切换到 DeepSeek 官方推理模型
根因：
- 现有默认模型仍是 OpenRouter 的 `tngtech/deepseek-r1t2-chimera:free`。
- 统一模型配置抽象不足，不利于稳定切换提供商。

## 3. 实施方案与落地状态

### A. 历史引用修复（已完成）
1. 会话消息结构升级：
- `rag.py` 的 `ConversationMessage` 增加 `references/metadata/agent_type`。
2. 持久化引用：
- `rag.py /ask` 保存 assistant 消息时写入 references + metadata。
- `agents.py /stream` 保存 assistant 消息时写入 references + metadata。
3. 历史接口返回完整消息：
- `GET /rag/conversations`、`GET /rag/conversations/{id}` 均返回完整字段。
4. 前端历史回放恢复引用：
- 选中历史会话后，从最近一条 assistant 消息提取 references 显示。

### B. 引用可读性优化（已完成）
1. 统一标题回填：
- `rag.py` 与 `agents.py` 新增批量标题补齐逻辑：按 `paper_id` 查询 `Paper.title`。
2. 前端右侧来源面板增强：
- 新增相关度分数标签显示（score）。

### C. 答案质量与展示样式优化（已完成）
1. Prompt 结构化约束增强：
- `backend/app/rag/prompts.py` 增加固定输出章节要求（结论/依据/局限/参考来源）。
2. 非检索 Agent 输出归一化：
- `agents.py` 新增 `_normalize_agent_markdown`，将无结构文本整理为标准 Markdown 结构。
3. Chat 页面样式优化：
- `frontend/src/pages/Chat/index.css` 增强消息容器层次、引用卡片样式与移动端适配。

### D. 模型切换 DeepSeek 官方推理模型（已完成）
1. 新增统一配置：
- `LLM_API_KEY` / `LLM_BASE_URL` / `LLM_MODEL`
- 默认：`https://api.deepseek.com` + `deepseek-reasoner`
2. 保留旧配置兼容：
- `OPENROUTER_*` 继续可用，作为回退。
3. 统一调用入口：
- `engine.py` 与 skills 统一改为 `settings.EFFECTIVE_LLM_*`。
4. 已按需求写入本地 `backend/.env`（密钥未在文档中明文披露）。

### E. skills 缺陷与建议
已修复：
- 多处 skill 的模型参数写法不一致问题，统一为 `api_key/base_url` 与 `EFFECTIVE_LLM_*`。

建议后续迭代（未在本轮引入新依赖）：
1. 新增 `answer_citation_validator`：自动检查回答中 `[n]` 与 references 映射一致性。
2. 新增 `response_style_enforcer`：按任务类型（检索/综述/分析）输出固定骨架。
3. 合并或重命名 `utility_skills.py` 与 `util_skills.py`，降低维护歧义。
