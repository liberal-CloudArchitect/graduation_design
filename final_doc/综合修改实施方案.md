## 一、现状诊断总结

### 1.1 代码现状校准（基于实际代码审查）

> **重要**: 以下判断基于对 `backend/app/` 代码库的实际审查，而非计划文档描述。

| 模块 | 计划能力 | 实际代码能力 | 差距等级 | 说明 |
|------|---------|------------|---------|------|
| PDF 解析 | LayoutLMv3 智能布局 | PDFPlumber + OCR + LayoutAnalyzer 多层框架 | **严重** | 框架存在，但缺端到端 Markdown/LaTeX 恢复能力 |
| 元数据提取 | LLM + 布局感知 | 正则 + `LLMMetadataExtractor`（已实现但可选） | 中 | LLM 路径存在，可按需启用 |
| 复杂 PDF（表格/公式） | 完整识别与还原 | 不支持 | **严重** | 当前无法还原学术论文中的公式与复杂表格 |
| 检索策略 | BM25 + Dense + Rerank | **BM25 (ES) + Dense (Milvus) + RRF 融合已实现** | 中 | 骨架已跑通（`retriever.py` `HybridRetriever`），需做质量强化 |
| 文本切分 | 语义切分 / 父子索引 | `SemanticChunker` + `OverlapChunker`，无父子结构 | 高 | 切分器存在，但缺层级索引 |
| Agent 编排 | 动态任务分解 + 并发 | 固定角色路由 + `asyncio.gather` 并发 | 中高 | 缺动态子任务与状态流 |
| 上下文治理 | 中间件制度化 | 逻辑分散在 API/RAG/Agent 各层 | 中 | 缺统一治理框架 |

### 1.2 问题根因（校准后）

1. **"垃圾进，垃圾出"**: PDF 解析无法处理双栏/表格/公式，导致 RAG 喂给 LLM 的数据质量低下 — 这是全链路天花板
2. **检索质量可提升**: 已有 BM25+Dense+RRF 混合检索骨架，但缺少检索质量诊断、稀疏向量增强、专业重排模型
3. **切片过碎**: 简单分块破坏了章节级逻辑关联，LLM 看到的是断章取义的碎片
4. **编排粒度粗**: 复杂问题无法动态分解为子任务并行探索，覆盖维度受限

### 1.3 应保留的核心资产

以下已实现的能力经评估质量较好，融合升级时**必须保留**：

- **BGE-M3 向量化 + Milvus 存储**: 基础向量化流程已跑通
- **BM25 + Dense + RRF 混合检索**: `retriever.py` 中 `HybridRetriever` 已实现并行双路检索与 RRF 融合
- **RAG 证据治理链**: `engine.py` 中的噪声过滤、参考文献识别、低信号过滤、文献多样性机制
- **记忆体系**: DynamicMemory + ReconstructiveMemory + CrossMemory 组合高于常见方案
- **API 契约**: `/api/v1/rag/*`, `/api/v1/agent/*` 等前端已对接的接口
- **多存储互补架构**: Postgres/Mongo/Milvus/ES/Redis 职责边界清晰

## 二、总体升级策略

### 2.1 三轴升级模型

```
┌──────────────────────────────────────────────────────────────┐
│                      输出质量提升                               │
├────────────────┬──────────────────────┬──────────────────────┤
│  轴1: 数据质量   │  轴2: 检索与索引      │  轴3: 编排与推理       │
│  (源头治理)      │  (引擎增强)           │  (能力深化)           │
├────────────────┼──────────────────────┼──────────────────────┤
│ MinerU 端到端   │ 现有 BM25+RRF 强化    │ Runtime 中间件框架     │
│ PDF 解析        │ + BGE sparse 实验分支  │                      │
├────────────────┼──────────────────────┼──────────────────────┤
│ 智能解析路由     │ 父子文档索引           │ 动态子任务执行器       │
│ (简单/复杂分流)  │ (小块检索大块生成)     │                      │
├────────────────┼──────────────────────┼──────────────────────┤
│ 结构化 Markdown  │ BGE-Reranker 重排序   │ 任务状态事件流        │
│ 输出 + 降级链    │ 升级                  │                      │
├────────────────┼──────────────────────┼──────────────────────┤
│                │ GraphRAG 知识图谱      │ 引用一致性校验         │
│                │ (P2 增强层)           │                      │
└────────────────┴──────────────────────┴──────────────────────┘
```

### 2.2 实施原则

1. **先基线后改造**: 任何改造开始前，先固化评测基线，防止"改完后无法证明收益"
2. **内核不替换**: 保留 `engine.py` 为事实证据主引擎，升级而非重写
3. **源头优先**: PDF 解析质量是整个链路天花板，优先解决
4. **接口保持兼容**: 对前端 API 响应结构不做破坏性变更
5. **分阶段可回滚**: 每阶段通过 Feature Flag 控制，不做一次性切换
6. **现有路径不拆除**: 已实现的 ES BM25 路径保留为稳定主路径，新增能力以实验分支并存

## 三、评测方法论与指标定义

> 本节为后续各阶段验收标准提供统一的评测口径，避免"目标值无数据集绑定"的争议。

### 3.1 评测数据集定义

| 数据集 | 用途 | 规模 | 构成 | 来源 |
|-------|------|------|------|------|
| **ParseBench** | PDF 解析质量评测 | ≥ 20 篇 | 含双栏、公式密集、扫描件、表格密集各 ≥ 5 篇 | 从已上传文献 + 公开学术 PDF 手工选取 |
| **RetrievalBench** | 检索质量评测 | ≥ 50 条 query-relevance 标注 | 涵盖精确术语、语义模糊、跨文献 | 基于现有 `tests/llm_eval_results` 扩展 |
| **AnswerBench** | 回答质量评测 | ≥ 30 条 | 简单事实(10) + 跨段落(10) + 复杂多步(10) | 人工构造，含标准答案 |
| **LatencyBench** | 性能基准 | 同上 query 集 | N/A | 自动化执行 |

### 3.2 指标定义与统计方法

| 指标 | 定义 | 计算方式 | 统计口径 |
|-----|------|---------|---------|
| **解析正确率** | 抽样页面中结构还原正确的比例 | 人工标注（正确/部分正确/错误），取正确+部分正确占比 | ParseBench，≥ 3 人交叉标注 |
| **Recall@K** | Top-K 检索结果中包含相关文档的比例 | 标准 IR Recall 公式 | RetrievalBench, K=10,20 |
| **nDCG@K** | 考虑排序位置的检索质量 | 标准 nDCG 公式 | RetrievalBench, K=10 |
| **引用一致性** | 回答中引用编号与实际证据匹配的比例 | 正确引用数 / 总引用数 | AnswerBench, 自动 + 人工复核 |
| **答案完整度** | 标准答案要点被覆盖的比例 | 覆盖要点数 / 标准答案要点总数 | AnswerBench, 人工评分 |
| **P95 延迟** | 95% 分位请求延迟 | 排序取 95% 分位 | LatencyBench, 连续 3 轮取均值 |

### 3.3 提升目标表述规范

后续所有"提升 X%"均指：**在同一评测集上，候选版本指标 vs 阶段 0 基线指标的百分点差值**。目标值为预期方向性参照，实际以评测结果为准，不作为硬性 pass/fail 唯一依据 — 核心门控为 **"不低于基线"**。

### 阶段 1: MinerU PDF 解析集成（P0 — 核心瓶颈，第 2-3 周）

**目标**: 用端到端多模态模型增强 PDF 解析能力，从源头保证数据质量。保留现有 PDFPlumber/OCR/LayoutAnalyzer 作为降级链。

**工期**: 1.5 ~ 2 周

#### 1.1 技术方案

MinerU 是基于 Vision-Language 模型的端到端 PDF 解析工具，直接输入 PDF 页面图像，输出结构化 Markdown/LaTeX。相比当前 PDFPlumber + LayoutAnalyzer 的多阶段 pipeline，MinerU 在复杂版面场景下显著减少中间误差累积。

**解析路由策略** — 并非所有 PDF 都需要重量级模型：

```
PDF 输入
  │
  ├─ 预检测（页数、是否扫描件、是否含公式/表格）
  │
  ├─ 简单 PDF（纯文本、单栏、无公式表格）
  │     → PyMuPDF 快速提取（毫秒级，CPU 即可）
  │
  └─ 复杂 PDF（双栏、扫描件、含公式/表格/图表）
        → MinerU 端到端解析（秒级/页，需 GPU 或 API）
        → 输出: 结构化 Markdown（含 LaTeX 公式、表格）
        │
        └─ MinerU 失败 → Layout + OCR → TextExtractor（降级链）
```

```python
class PDFParser:
    def __init__(self, ...):
        self.simple_parser = TextExtractor()       # PyMuPDF (保留)
        self.complex_parser = MinerUParser(...)     # MinerU (新增)
        self.layout_analyzer = LayoutAnalyzer()     # 降级路径 (保留)
        self.ocr_engine = OCREngine()               # 降级路径 (保留)
    
    async def parse(self, pdf_path: str) -> PDFDocument:
        if not settings.MINERU_ENABLED:
            return await self._parse_legacy(pdf_path)
        
        complexity = self._detect_complexity(pdf_path)
        if complexity == 'simple':
            return await self._parse_simple(pdf_path)
        else:
            return await self._parse_with_mineru_and_fallback(pdf_path)
    
    async def _parse_with_mineru_and_fallback(self, pdf_path: str) -> PDFDocument:
        """MinerU 解析，失败时降级到 Layout+OCR → TextExtractor"""
        try:
            return await asyncio.wait_for(
                self.complex_parser.parse(pdf_path),
                timeout=settings.PDF_PARSE_TIMEOUT
            )
        except Exception as e:
            logger.warning(f"MinerU failed, falling back: {e}")
            return await self._parse_legacy(pdf_path)
```

### 阶段 2: 父子文档索引与语义切分升级（P0 — 第 3-5 周）

**目标**: 解决切片过碎导致上下文丢失的问题，让 LLM 看到完整的逻辑段落

**工期**: 1.5 ~ 2 周（与阶段 1 部分并行）

#### 2.1 技术方案

**核心思路: 小切片检索，大窗口生成**

```
原文文档 (Markdown)
  │
  ├─ 父块 (Parent Chunk): 整个章节或小节
  │     用途: 送入 LLM 上下文，保证逻辑完整性
  │
  └─ 子块 (Child Chunk): 段落级细粒度切分
        用途: 向量化索引，保证检索精准性
        ↑ 每个子块维护 parent_id 指针
```

检索时命中子块 → 返回对应的父块给 LLM → LLM 看到完整章节上下文。

#### 2.2 实现步骤

**Step 1: 改造 SemanticChunker**

修改 `backend/app/rag/chunker.py`：

```python
@dataclass
class Chunk:
    text: str
    metadata: dict
    chunk_type: str = "child"     # "parent" | "child"
    parent_id: str = None         # 子块指向父块的 ID
    section_path: str = None      # 章节路径，如 "3.2.1 实验设置"

class HierarchicalChunker:
    """层级切分器: 生成父子文档结构"""
    
    def __init__(self, parent_max_tokens=2000, child_max_tokens=400):
        ...
    
    def chunk(self, markdown: str, paper_id: int) -> List[Chunk]:
        """
        1. 解析 Markdown 章节结构（H1/H2/H3 标题树）
        2. 每个最低层级小节 → 一个父块
        3. 父块内按段落/语义边界 → 多个子块
        4. 子块携带 parent_id 和 section_path
        """
        sections = self._parse_section_tree(markdown)
        chunks = []
        for section in sections:
            parent = self._create_parent_chunk(section, paper_id)
            children = self._split_to_children(section, parent.metadata['chunk_id'])
            chunks.append(parent)
            chunks.extend(children)
        return chunks
```

```python
# backend/app/runtime/middleware/base.py
class Middleware:
    async def before(self, state: ExecutionState) -> ExecutionState:
        return state
    
    async def after(self, state: ExecutionState, result: Dict) -> Dict:
        return result

class MiddlewareChain:
    def __init__(self, middlewares: List[Middleware]):
        self.middlewares = middlewares
    
    async def run(self, state: ExecutionState, handler) -> Dict:
        for mw in self.middlewares:
            state = await mw.before(state)
        result = await handler(state)
        for mw in reversed(self.middlewares):
            result = await mw.after(state, result)
        return result
```

```python
# backend/app/runtime/task_executor.py
class TaskExecutor:
    """动态子任务执行器 — 将复杂问题分解为可并行的子任务"""
    
    def __init__(self, max_concurrent: int = 3, timeout: int = 60):
        self.max_concurrent = max_concurrent
        self.timeout = timeout
        self.semaphore = asyncio.Semaphore(max_concurrent)
    
    async def execute(self, state: ExecutionState) -> Dict:
        if self._is_simple_query(state.query):
            return await self._direct_execute(state)
        
        subtasks = await self._plan_subtasks(state)
        
        results = await asyncio.gather(
            *[self._run_subtask(st) for st in subtasks],
            return_exceptions=True
        )
        
        return await self._synthesize(state, results)
    
    async def _run_subtask(self, subtask: SubtaskState) -> Dict:
        """带超时和信号量的子任务执行"""
        async with self.semaphore:
            subtask.status = TaskStatus.RUNNING
            try:
                result = await asyncio.wait_for(
                    self._dispatch(subtask),
                    timeout=self.timeout
                )
                subtask.status = TaskStatus.COMPLETED
                return result
            except asyncio.TimeoutError:
                subtask.status = TaskStatus.TIMED_OUT
                return {"error": "subtask timed out"}
```

```python
        project_id=project_id
    )
    
    merged = self._merge_results(vector_results, graph_results)
    return await self._rerank(query, merged, top_k=top_k)
```

## 七、模型选型总结

| 用途 | 当前模型 | 建议模型 | 变更原因 |
|-----|---------|---------|---------|
| PDF 解析 | PDFPlumber + LayoutAnalyzer | **MinerU (Qwen2VL-1.2B)** + 降级链 | 端到端多模态解析，还原公式/表格 |
| 文本嵌入 | BGE-M3 (dense only) | **BGE-M3 (不变)**，可选启用 sparse | 仍为 SOTA |
| 关键词检索 | Elasticsearch BM25 | **ES BM25 (保留)** | 已实现且稳定，保留为主路径 |
| 重排序 | 无专用模型 | **BGE-Reranker-v2-m3** | 支持多语言、长窗口 |
| LLM 生成 | ChatGLM3-6B / DeepSeek | **DeepSeek-V3 (或 API)** | 数理逻辑更强，学术场景更优 |
| 知识图谱 | 无 | **LLM 抽取 + Neo4j 存储** (P2) | 支持跨文档结构化推理 |

## 十一、待确认事项

在开始实施前，以下事项需要确认：

1. **GPU 资源**: MinerU 本地推理需要 4-8GB VRAM，是否有可用 GPU 服务器？若无，是否接受 API 模式或 Colab 预处理模式？
2. **Neo4j 部署**: GraphRAG (P2) 需要图数据库，是否接受在 Docker 中增加 Neo4j 容器？
3. **DeepSeek-V3 接入**: LLM 生成端是否切换到 DeepSeek-V3 API？还是继续使用现有模型？
4. **已索引数据迁移**: 父子索引升级后已有文献需重新索引；sparse 分支启用也需 backfill。是否接受一次性开销？
5. **前端适配**: Runtime 事件流（SSE）需要前端配合展示任务进度，是否有前端开发资源？
6. **评测标注人力**: 阶段 0 构建评测集需要人工标注 query-relevance 和标准答案，预计 1-2 人日。
7. **高价值样本**: 请提供当前最关心的 20 个高价值问题样本，用于构建评测集核心子集。
8. **引用格式**: 是否有标准引用格式要求（如 GB/T 7714、APA），用于引用一致性校验输出。
9. **语言优先级**: 是否优先支持中文、英文或双语（影响 embedding/rerank 参数与分词策略）。
