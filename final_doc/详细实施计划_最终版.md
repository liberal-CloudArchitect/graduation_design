#### PDF解析 (多层架构)

```python
# services/pdf_parser.py

class PDFParser:
    """多层PDF解析器"""
    
    def __init__(self):
        # Layer 1: 基础文本提取
        self.text_extractor = PDFPlumberExtractor()
        # Layer 2: OCR (扫描件)
        self.ocr_engine = TesseractOCR()
        # Layer 3: 布局分析 (复杂PDF) - 替代ChatGLM
        self.layout_analyzer = LayoutLMv3Analyzer()
        # Layer 4: 信息整合 (调用LLM API)
        self.info_extractor = LLMInfoExtractor()
    
    async def parse(self, pdf_path: str) -> dict:
        # Step 1: 尝试文本提取
        text = self.text_extractor.extract(pdf_path)
        
        # Step 2: 如果文本太少，用OCR
        if len(text) < 100:
            text = self.ocr_engine.recognize(pdf_path)
        
        # Step 3: 布局分析 (识别标题、段落、表格)
        layout = await self.layout_analyzer.analyze(pdf_path)
        
        # Step 4: 使用LLM整合信息
        metadata = await self.info_extractor.extract(text, layout)
        
        return metadata
```

#### LayoutLMv3 布局分析 (替代ChatGLM微调)

```python
# services/layout_analyzer.py
from transformers import LayoutLMv3Processor, LayoutLMv3ForTokenClassification

class LayoutLMv3Analyzer:
    """布局分析器 - 替代ChatGLM用于PDF解析"""
    
    def __init__(self):
        self.processor = LayoutLMv3Processor.from_pretrained(
            "microsoft/layoutlmv3-base"
        )
        self.model = LayoutLMv3ForTokenClassification.from_pretrained(
            "microsoft/layoutlmv3-base",
            num_labels=7  # title, author, abstract, keywords, section, table, figure
        )
    
    async def analyze(self, pdf_path: str) -> dict:
        """分析PDF布局结构"""
        # 转换PDF为图像
        images = pdf_to_images(pdf_path)
        
        # 对每页进行布局分析
        results = []
        for image in images:
            inputs = self.processor(image, return_tensors="pt")
            outputs = self.model(**inputs)
            # 解析布局标签
            layout = self._parse_layout(outputs)
            results.append(layout)
        
        return self._merge_layouts(results)
```

```python
# app/rag/qa_engine.py
from langchain_openai import ChatOpenAI

class RAGQAEngine:
    """RAG问答引擎 - MVP版本"""
    
    def __init__(self):
        # 使用原版BGE-M3 (不微调)
        self.embedder = BGEM3FlagModel('BAAI/bge-m3')
        # 使用云端LLM API
        self.llm = ChatOpenAI(model="gpt-4o-mini")
        # 向量数据库
        self.vector_store = MilvusClient()
    
    async def answer(self, question: str, project_id: int) -> dict:
        """回答问题"""
        # 1. 向量化问题
        query_embedding = self.embedder.encode(question)['dense_vecs']
        
        # 2. 检索相关文档
        docs = await self.vector_store.search(
            query_embedding, 
            filter={"project_id": project_id},
            top_k=5
        )
        
        # 3. 生成答案
        context = self._build_context(docs)
        answer = await self.llm.ainvoke(
            f"根据以下文献回答问题:\n{context}\n\n问题: {question}"
        )
        
        return {
            "answer": answer.content,
            "references": docs,
            "method": "rag_mvp"  # 标记为MVP版本
        }
```

## 技术决策记录

### 决策1: 用LayoutLMv3替代ChatGLM进行PDF解析

| 对比项 | ChatGLM3-6B | LayoutLMv3 |
|-------|------------|-----------|
| 模型大小 | 12GB | 400MB |
| 任务适配 | 语言生成 | **布局理解** ✅ |
| 推理速度 | 慢 | 快 |
| 显存需求 | 16GB+ | 4GB |

**结论**: LayoutLMv3更适合PDF布局分析任务，ChatGLM用于后续信息整合。

### 决策2: BGE-M3先基线再微调

| 策略 | 优点 | 缺点 |
|-----|------|------|
| 直接微调 | - | 可能灾难性遗忘 |
| **先基线后微调** | 风险可控，节省成本 | 需额外评估时间 |

**结论**: 采用"先测后调"策略，Recall@10>85%则跳过微调。

### 决策3: RAG-First开发策略

| 策略 | 优点 | 缺点 |
|-----|------|------|
| 传统MVP→后补RAG | - | 架构重构，产品落后 |
| **RAG-First** | 产品竞争力强，架构统一 | 初期复杂度高 |

**结论**: MVP阶段即包含RAG问答，不在TF-IDF上浪费时间。
