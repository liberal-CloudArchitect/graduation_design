# LiterAI 文献分析平台 — 综合修改实施方案

> **版本**: v1.0  
> **日期**: 2026-03-03  
> **目标**: 从数据源头质量、检索引擎能力、智能体编排深度三个维度全面提升平台输出质量  
> **基线**: 当前 MVP 代码库（`backend/app/`）

---

## 一、现状诊断总结

### 1.1 核心差距矩阵

| 模块 | 计划能力 | 实际能力 | 差距等级 | 影响范围 |
|------|---------|---------|---------|---------|
| PDF 解析 | LayoutLMv3 智能布局分析 | PDFPlumber 纯文本提取 | **严重** | 全链路数据质量 |
| 元数据提取 | LLM + 布局感知 | 正则表达式匹配 | 高 | 文献管理准确性 |
| 复杂 PDF（表格/公式） | 完整识别与还原 | 不支持 | **严重** | 学术文献覆盖率 |
| 检索策略 | BM25 + Dense + Rerank | 仅 Dense 向量检索 | 高 | 召回率与精度 |
| 文本切分 | 语义切分 / 父子索引 | 简单固定长度分块 | 高 | 上下文完整性 |
| Agent 编排 | 动态任务分解 + 并发 | 固定角色路由 + 简单并发 | 中高 | 复杂任务处理深度 |
| 上下文治理 | 中间件制度化 | 逻辑分散在各层 | 中 | 系统稳定性与可维护性 |

### 1.2 问题根因

1. **"垃圾进，垃圾出"**: PDF 解析能力极弱，双栏/表格/公式全部丢失或乱序，导致 RAG 喂给 LLM 的数据本身质量低下
2. **检索单一**: 仅有稠密向量检索，缺少关键词匹配能力，对精确术语查询召回率不足
3. **切片过碎**: 简单分块破坏了章节级逻辑关联，LLM 看到的是断章取义的碎片
4. **编排粒度粗**: 复杂问题无法动态分解为子任务并行探索，覆盖维度受限

### 1.3 应保留的核心资产

以下已实现的能力经评估质量较好，融合升级时**必须保留**：

- **BGE-M3 向量化 + Milvus 存储**: 基础向量化流程已跑通
- **RAG 证据治理链**: `engine.py` 中的噪声过滤、参考文献识别、低信号过滤机制
- **记忆体系**: DynamicMemory + ReconstructiveMemory + CrossMemory 组合高于常见方案
- **API 契约**: `/api/v1/rag/*`, `/api/v1/agent/*` 等前端已对接的接口
- **多存储互补架构**: Postgres/Mongo/Milvus/ES/Redis 职责边界清晰

## 二、总体升级策略

### 2.1 三轴升级模型

```
┌─────────────────────────────────────────────────────────┐
│                    输出质量提升                            │
├───────────────┬───────────────────┬─────────────────────┤
│  轴1: 数据质量  │  轴2: 检索与索引   │  轴3: 编排与推理     │
│  (源头治理)     │  (引擎增强)        │  (能力深化)          │
├───────────────┼───────────────────┼─────────────────────┤
│ MinerU 端到端  │ BGE-M3 混合检索    │ Runtime 中间件框架   │
│ PDF 解析       │ (Sparse+Dense)    │                     │
├───────────────┼───────────────────┼─────────────────────┤
│ 智能解析路由    │ 父子文档索引       │ 动态子任务执行器      │
│ (简单/复杂分流) │ (小块检索大块生成)  │                     │
├───────────────┼───────────────────┼─────────────────────┤
│ 结构化 Markdown │ BGE-Reranker      │ 任务状态事件流       │
│ 输出           │ 重排序             │                     │
├───────────────┼───────────────────┼─────────────────────┤
│               │ GraphRAG 知识图谱   │ 引用一致性校验       │
│               │ (增强层)           │                     │
└───────────────┴───────────────────┴─────────────────────┘
```

### 2.2 实施原则

1. **内核不替换**: 保留 `engine.py` 为事实证据主引擎，升级而非重写
2. **源头优先**: PDF 解析质量是整个链路天花板，优先解决
3. **接口保持兼容**: 对前端 API 响应结构不做破坏性变更
4. **分阶段可回滚**: 每阶段通过 Feature Flag 控制，不做一次性切换
5. **逐步迁移**: 先增强数据层，再升级检索层，最后深化编排层

## 三、分阶段实施方案

### 阶段 1: MinerU PDF 解析集成（P0 — 核心瓶颈）

**目标**: 用端到端多模态模型替代当前 PDFPlumber 纯文本提取，从源头保证数据质量

**工期**: 1.5 ~ 2 周

#### 1.1 技术方案

MinerU 是基于 Vision-Language 模型的端到端 PDF 解析工具，直接输入 PDF 页面图像，输出结构化 Markdown/LaTeX。相比 PDFPlumber + LayoutLMv3 的多阶段 pipeline，MinerU 显著减少中间误差累积。

**解析路由策略** — 并非所有 PDF 都需要重量级模型：

```
PDF 输入
  │
  ├─ 预检测（页数、是否扫描件、是否含公式/表格）
  │
  ├─ 简单 PDF（纯文本、单栏、无公式表格）
  │     → PyMuPDF 快速提取（毫秒级，CPU 即可）
  │
  └─ 复杂 PDF（双栏、扫描件、含公式/表格/图表）
        → MinerU 端到端解析（秒级/页，需 GPU 或 API）
        → 输出: 结构化 Markdown（含 LaTeX 公式、表格）
```

### 阶段 2: 父子文档索引与语义切分升级（P0 — 与阶段 1 协同）

**目标**: 解决切片过碎导致上下文丢失的问题，让 LLM 看到完整的逻辑段落

**工期**: 1 ~ 1.5 周（可与阶段 1 部分并行）

#### 2.1 技术方案

**核心思路: 小切片检索，大窗口生成**

```
原文文档 (Markdown)
  │
  ├─ 父块 (Parent Chunk): 整个章节或小节
  │     用途: 送入 LLM 上下文，保证逻辑完整性
  │
  └─ 子块 (Child Chunk): 段落级细粒度切分
        用途: 向量化索引，保证检索精准性
        ↑ 每个子块维护 parent_id 指针
```

检索时命中子块 → 返回对应的父块给 LLM → LLM 看到完整章节上下文。

**Step 1: 改造 SemanticChunker**

    metadata: dict
    chunk_type: str = "child"     # "parent" | "child"
    parent_id: str = None         # 子块指向父块的 ID

class HierarchicalChunker:
    """层级切分器: 生成父子文档结构"""
    
    def __init__(self, parent_max_tokens=2000, child_max_tokens=400):
        ...
    
    def chunk(self, markdown: str, paper_id: int) -> List[Chunk]:
        """
        1. 解析 Markdown 章节结构（H1/H2/H3 标题树）
        2. 每个最低层级小节 → 一个父块
        3. 父块内按段落/语义边界 → 多个子块
        4. 子块携带 parent_id 和 section_path

        sections = self._parse_section_tree(markdown)
        chunks = []
        for section in sections:
            parent = self._create_parent_chunk(section, paper_id)
            children = self._split_to_children(section, parent.metadata['chunk_id'])
            chunks.append(parent)
            chunks.extend(children)
        return chunks

**Step 3: 实现 RRF 混合检索**

在 `backend/app/rag/retriever.py` 中新增混合检索逻辑：

```python
class HybridRetriever:
    """BGE-M3 原生 Sparse+Dense 混合检索器"""
    
    async def search(self, query: str, top_k: int = 20, 
                     project_id: int = None) -> List[RetrievalResult]:
        query_embedding = self.engine.embed([query])
        
        # 双路并行检索
        dense_results, sparse_results = await asyncio.gather(
            self._dense_search(query_embedding['dense'][0], top_k * 2),
            self._sparse_search(query_embedding['sparse'][0], top_k * 2)
        )
        
        # RRF (Reciprocal Rank Fusion) 融合
        return self._rrf_fusion(dense_results, sparse_results, k=60, top_k=top_k)
    
    def _rrf_fusion(self, dense_results, sparse_results, k=60, top_k=20):
        """RRF 融合算法:
        score(d) = Σ 1/(k + rank_i(d))
        k=60 为标准超参数，平衡两路结果权重
        """
        scores = {}
        for rank, result in enumerate(dense_results):
            scores[result.id] = scores.get(result.id, 0) + 1.0 / (k + rank + 1)
        for rank, result in enumerate(sparse_results):
            scores[result.id] = scores.get(result.id, 0) + 1.0 / (k + rank + 1)
        
        sorted_ids = sorted(scores, key=scores.get, reverse=True)[:top_k]
        return [self._get_result(id) for id in sorted_ids]
```

**Step 4: 增加 BGE-Reranker 重排序**

```python
async def _rerank(self, query: str, docs: List[Dict], top_k: int = 5):
    """使用 BGE-Reranker-v2-m3 对候选文档重排序"""
    if not docs:
        return []
    
    pairs = [(query, doc['text']) for doc in docs]
    scores = self.reranker.compute_score(pairs)
    
    ranked = sorted(zip(docs, scores), key=lambda x: x[1], reverse=True)
    return [doc for doc, _ in ranked[:top_k]]
```

#### 4.3 中间件链

```python
# backend/app/runtime/middleware/base.py
class Middleware:
    async def before(self, state: ExecutionState) -> ExecutionState:
        return state
    
    async def after(self, state: ExecutionState, result: Dict) -> Dict:
        return result

# 中间件链执行器
class MiddlewareChain:
    def __init__(self, middlewares: List[Middleware]):
        self.middlewares = middlewares
    
    async def run(self, state: ExecutionState, handler) -> Dict:
        for mw in self.middlewares:
            state = await mw.before(state)
        result = await handler(state)
        for mw in reversed(self.middlewares):
            result = await mw.after(state, result)
        return result
```

#### 4.4 动态子任务执行器

```python
# backend/app/runtime/task_executor.py
class TaskExecutor:
    """动态子任务执行器 — 将复杂问题分解为可并行的子任务"""
    
    def __init__(self, max_concurrent: int = 3, timeout: int = 60):
        self.max_concurrent = max_concurrent
        self.timeout = timeout
        self.semaphore = asyncio.Semaphore(max_concurrent)
    
    async def execute(self, state: ExecutionState) -> Dict:
        # 1. 判断是否需要分解（简单问题直走单 agent）
        if self._is_simple_query(state.query):
            return await self._direct_execute(state)
        
        # 2. LLM 规划子任务
        subtasks = await self._plan_subtasks(state)
        
        # 3. 并发执行（含超时与并发限制）
        results = await asyncio.gather(
            *[self._run_subtask(st) for st in subtasks],
            return_exceptions=True
        )
        
        # 4. 综合结果
        return await self._synthesize(state, results)
    
    async def _plan_subtasks(self, state: ExecutionState) -> List[SubtaskState]:
        """LLM 分析查询复杂度，规划子任务列表"""
        ...
    
    async def _run_subtask(self, subtask: SubtaskState) -> Dict:
        """带超时和信号量的子任务执行"""
        async with self.semaphore:
            subtask.status = TaskStatus.RUNNING
            try:
                result = await asyncio.wait_for(
                    self._dispatch(subtask),
                    timeout=self.timeout
                )
                subtask.status = TaskStatus.COMPLETED
                return result
            except asyncio.TimeoutError:
                subtask.status = TaskStatus.TIMED_OUT
                return {"error": "subtask timed out"}
```

**学术场景专用子任务模板**:

| 子任务类型 | 触发场景 | 执行策略 |
|-----------|---------|---------|
| `retrieval_deep_dive` | 需要从多篇文献深挖某一主题 | 扩大 top_k、跨文献聚合 |
| `method_comparison` | "比较 A 和 B 方法的优劣" | 分别检索 A 和 B 相关证据，对比分析 |
| `evidence_checker` | 校验已有结论的证据支撑度 | 反向检索反例和支持证据 |
| `synthesis_writer` | 综述/总结类问题 | 汇总子任务结果，结构化输出 |

```python
async def enhanced_search(self, query: str, project_id: int, top_k: int = 10):
    """向量检索 + 图谱检索双路融合"""
    
    # 路径 1: 向量混合检索（现有能力）
    vector_results = await self.hybrid_retriever.search(query, top_k=top_k * 3)
    
    # 路径 2: 图谱检索（新增能力）
    graph_entities = await self._extract_query_entities(query)
    graph_results = await self.kg_retriever.search(
        entities=graph_entities,
        hops=2,  # 2 跳关联
        project_id=project_id
    )
    
    # 融合 + 重排序
    merged = self._merge_results(vector_results, graph_results)
    return await self._rerank(query, merged, top_k=top_k)
```

| 阶段 4 (Runtime) | 复杂任务深度 | 多维度证据覆盖 +25%~60%，引用一致性错误 -20%~40% |
| 阶段 5 (GraphRAG) | 跨文档推理 | 可回答结构化关系查询，综述类问题质量提升 |

### 9.2 整体目标

完成全部阶段后，平台将从 **"基础文档阅读器"** 升级为 **"能稳定分解复杂学术任务并输出高质量证据化答案的科研智能分析平台"**：

- 简单问答: 体验基本持平或小幅改善
- 复杂研究任务（比较、综述、争议分析、多来源整合）: 质量与可解释性显著提升
- 工程侧: 从"功能堆叠"转向"运行时治理"，后续迭代成本下降

## 十、待确认事项

在开始实施前，以下事项需要确认：

1. **GPU 资源**: MinerU 本地推理需要 4-8GB VRAM，是否有可用 GPU 服务器？若无，是否接受 API 模式或 Colab 预处理模式？
2. **Neo4j 部署**: GraphRAG 需要图数据库，是否接受在 Docker 中增加 Neo4j 容器？
3. **DeepSeek-V3 接入**: LLM 生成端是否切换到 DeepSeek-V3 API？还是继续使用现有模型？
4. **已索引数据迁移**: 混合检索和父子索引升级后，已有文献是否需要重新索引？（建议是，但会有一次性开销）
5. **前端适配**: Runtime 事件流（SSE）需要前端配合展示任务进度，是否有前端开发资源？
