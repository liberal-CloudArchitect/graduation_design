# LiterAI 综合项目修改实施方案（架构优化与输出质量提升）

  


## 1. 方案目标与约束

### 1.1 总体目标

在不破坏现有业务 API 契约的前提下，完成两类能力升级并形成统一架构：

1. 数据入口质量升级：将 PDF 解析从“文本抽取优先”升级为“复杂文档端到端结构化优先”，重点解决公式、表格、双栏乱序问题。

2. 运行时执行能力升级：将固定角色路由升级为可治理的运行时编排（中间件 + 子任务执行器 + 事件流 + 技能扩展）。

3. 检索与证据链升级：强化父子索引、混合检索、重排与引用一致性，提升回答稳定性和可追溯性。

4. 可持续演进：建立 Feature Flag、评测基线、灰度与回滚机制，避免“改架构即降质”。

### 1.2 必须满足的硬约束

1. 保留 `backend/app/rag/engine.py` 的学术证据治理主链路，不做推倒重来。

2. 保持 `/api/v1/rag/*/api/v1/agent/*` 接口兼容。

3. 严格保持 `project_id/user_id` 维度的数据隔离。

4. 所有新增能力必须可开关、可灰度、可回滚。

## 2. 两份方案的综合结论（冲突校准后）

两份输入文档的主张可以整合为“证据内核不替换 + 执行底座增强”的路线，但存在现状判断差异。基于当前代码结构，综合判断如下：

1. PDF 解析：当前已包含 `PDFPlumber + OCR + LayoutAnalyzer` 多层框架，但缺少对复杂学术版面的端到端 Markdown/LaTeX 恢复能力，MinerU 集成属于 P0。

2. 检索：当前代码中已有 `BM25 + Vector + RRF` 混合检索骨架，重点不是“从 0 到 1”，而是“质量强化与稀疏向量增强（可选 BGE-M3 sparse）”。

3. 编排：当前仍以 `agents/coordinator.py` 为固定角色路由，缺少 deer-flow 风格的运行时中间件治理与任务状态流，这部分是结构升级核心。

4. GraphRAG：价值明确，但应后置为增强项（P2），先保证解析与执行链稳定。

## 3. 融合后目标架构

```text

Frontend / API (保持现有契约)

  -> /api/v1/rag | /api/v1/agent

  -> Runtime Orchestration Layer (新增)

     -> Middleware Chain (clarify / summarize / upload / budget / observability)

     -> Task Executor (dynamic subtasks + concurrency guardrails)

     -> Skill Resolver (code skills + markdown skills)

     -> Tool Hub (local tools + MCP tools)

  -> Academic Evidence Core (保留并增强)

     -> Parser Router (Simple Parser / MinerU)

     -> Parent-Child Index + Hybrid Retrieval + Rerank

     -> Citation Grounding + Consistency Validator

     -> Project Memory (dynamic/reconstructive/cross)

  -> Storage (Postgres / MongoDB / Milvus / Elasticsearch / Redis)

```

## 4. 分阶段实施路线（12 周）

## 阶段0：基线固化与差异对齐（第1周）

目标：防止后续改造“看起来先进、实际退化”。

关键任务：

1. 固化评测集与指标口径（Recall@K、nDCG@K、citation precision、事实一致性、P95 延迟、失败率）。

2. 输出“代码现状校准清单”：明确已实现与未实现项（尤其是混合检索、layout 能力、rerank 实际路径）。

3. 建立候选版本对照脚本（baseline vs candidate）。

交付物：

1. `docs/architecture_baseline_2026Q1.md`

2. `backend/tests/eval_fusion_baseline/`

验收标准：

1. 同一数据集可重复出报表。

2. 指标计算脚本固定且可 CI 运行。

---

## 阶段1：P0 解析链升级（MinerU 集成 + 解析路由）（第2-4周）

目标：优先提升输入数据质量，这是输出质量上限的决定项。

关键任务：

1. 在 `PDFParser` 增加解析路由器：

   - 简单文档：沿用 `PDFPlumber/PyPDF` 快路。

   - 复杂文档（扫描、双栏、公式密集）：走 MinerU。

2. 新增 `MinerUParser` 适配层，统一输出结构：

   - page_text

   - markdown_block

   - formula/table 标记

   - 版面元数据（页码、bbox 可选）

3. 增加失败降级链：

   - MinerU 失败 -> Layout + OCR -> TextExtractor

4. 引入解析质量评分（可解释规则）：

   - 乱码率、公式保真率、表格还原率、阅读顺序一致性。

建议改动位置：

1. `backend/app/services/pdf_parser.py`

2. `backend/app/services/mineru_parser.py`（新增）

3. `backend/app/services/layout_analyzer.py`（接口对齐）

4. `backend/app/api/v1/papers.py`（异步入库任务接线）

Feature Flags：

1. `PARSER_ROUTER_ENABLED`

2. `MINERU_ENABLED`

验收标准：

1. 复杂论文集解析成功率 >= 95%。

2. 公式/表格场景的人工抽检正确率较基线提升 >= 25%。

3. 单篇处理失败可自动降级并产生日志可追踪。

---

## 阶段2：P0 索引结构升级（Parent-Child + 引用锚点）（第3-5周）

目标：解决“检索命中精准但上下文碎片化”问题。

关键任务：

1. 设计父子块结构：

   - 父块：章节/逻辑段（供生成）

   - 子块：句群/段落（供检索）

2. Milvus/Mongo 存储补充字段：

   - `parent_idsection_titlechunk_typecitation_anchor`。

3. 检索命中子块后自动回填父块上下文。

4. 引用映射升级：答案引用编号可稳定回溯到父子块与页码。

建议改动位置：

1. `backend/app/rag/engine.py`

2. `backend/app/rag/retriever.py`

3. `backend/app/services/semantic_chunker.py`（若存在，需扩展）

验收标准：

1. 跨段落问题的答案完整度提升 >= 20%。

2. 引用跳转准确率 >= 95%。

---

## 阶段3：P1 检索增强（Hybrid 强化 + Rerank 升级）（第4-6周）

目标：在已有混合检索基础上做质量强化，而非重复造轮子。

关键任务：

1. 保持现有 BM25+Dense+RRF 主路径，增加检索质量诊断日志（召回来源、融合得分、被过滤原因）。

2. 可选接入 BGE-M3 sparse 向量分支（实验开关）：

   - 方案A：Milvus sparse 字段与融合。

   - 方案B：维持 BM25 稳定路径，sparse 仅做 AB。

3. 重排模型升级评估`BGE-Reranker-v2-m3` 或当前 LLM-rerank 对照）。

4. 增加检索预算策略（按问题复杂度动态 top_k/重排窗口）。

Feature Flags：

1. `HYBRID_RETRIEVAL_V2_ENABLED`

2. `SPARSE_RETRIEVAL_ENABLED`

3. `RERANKER_V2_ENABLED`

验收标准：

1. Recall@20 提升 >= 10%。

2. 引用相关性误引率下降 >= 20%。

3. P95 延迟增幅 <= 20%。

---

## 阶段4：P1 运行时融合（Middleware + Task Executor + 事件流）（第5-9周）

目标：吸收 deer-flow 的执行治理能力，解决复杂任务深度不足与可观测性不足。

关键任务：

1. 新增 Runtime 层`backend/app/runtime/`）：

   - `state.py`

   - `runner.py`

   - `middleware/*`（clarification、summarization、upload、subtask_limit、observability）

2. 新增动态子任务执行器：

   - 状态`pending/running/completed/failed/timed_out`

   - 并发限制与超时双护栏

   - 禁递归策略（子任务不可再次无限分解）

3. API 接入：

   - `agents.py` 改为通过 runtime 调度

   - SSE 输出任务事件流（task_started/running/completed/failed）

4. 子任务模板化（学术场景）：

   - `retrieval_deep_dive`

   - `method_comparison`

   - `evidence_checker`

   - `synthesis_writer`

Feature Flags：

1. `RUNTIME_MIDDLEWARE_ENABLED`

2. `SUBTASK_MODE_ENABLED`

验收标准：

1. 复杂问题证据覆盖维度提升 >= 25%。

2. 超时率不高于基线 +3%。

3. 任务状态可在前端完整展示。

---

## 阶段5：P1 技能双轨与 MCP 扩展（第7-10周）

目标：保留现有代码技能稳定性，同时引入协议化扩展能力。

关键任务：

1. 双轨技能：

   - 轨道A：保留 `SkillRegistry` 代码技能。

   - 轨道B：新增 `MarkdownSkillLoader`（[SKILL.md](http://SKILL.md) 渐进加载）。

2. 工具网关：

   - MCP 配置化管理（stdio/http/sse）

   - 白名单、参数校验、审计日志。

3. 选择策略：

   - 优先命中代码技能；复杂流程任务才加载文档技能。

Feature Flags：

1. `MARKDOWN_SKILLS_ENABLED`

2. `MCP_TOOLS_ENABLED`

验收标准：

1. 新增流程型技能无需修改核心编排代码即可上线。

2. 外部工具故障不影响核心问答链路（可降级）。

---

## 阶段6：P2 GraphRAG 增强（第9-12周，可并行试点）

目标：提升跨论文关系推理与综述能力。

关键任务：

1. 先做“低成本冷启动”：

   - 仅抽取摘要+结论构图。

2. 图谱实体关系标准化：

   - 实体：METHOD/DATASET/METRIC/MODEL

   - 关系：USES/IMPROVES/EVALUATES_ON

3. 检索融合：

   - 向量召回 + 图谱跳数召回 + 统一重排。

说明：

1. 该阶段严格后置，必须在阶段1-5稳定后开启。

验收标准：

1. “方法比较/演化路径”类问题的回答完整性提升 >= 20%。

2. Token 成本在预算内可控（设置上限与早停）。

## 5. 质量保障与发布治理

### 5.1 质量门槛（硬指标）

1. 检索/回答质量不得低于阶段0基线。

2. 关键 API 兼容性必须 100%。

3. P95 延迟增幅 <= 20%。

4. 引用一致性错误率至少下降 30%（阶段5后）。

### 5.2 发布策略

1. 全程 Feature Flag 灰度。

2. 采用 Canary（10% -> 30% -> 100%）逐步放量。

3. 任一核心指标跌破门槛，自动回滚到上一稳定版本。

### 5.3 观测与审计

1. 子任务级 tracing（耗时、token、工具调用、失败原因）。

2. 检索级审计（召回来源、融合得分、最终入选原因）。

3. 引用级审计（答案片段 -> chunk -> 页码映射）。

## 6. 资源与分工建议

1. 后端架构/Runtime：1 人

2. RAG/检索与索引：1 人

3. 解析链（MinerU + PDF）：1 人

4. 测试评估与看板：1 人

最小可行团队：3-4 人。

## 7. 你已提供资源的使用建议（MinerU）

你已将 MinerU 模型目录放入工作区`MinerU2.5-2509-1.2B/`），可直接用于本地或容器推理适配。  

建议优先完成以下两项工程动作：

1. 建立 `MinerUParser` 适配层并打通到 `PDFParser` 路由。

2. 补充推理环境检查与降级逻辑（依赖缺失、显存不足、超时）。

## 8. 仍需你补充的关键信息

为确保实施阶段不返工，请补充以下内容：

1. 目标部署形态：本地单机、单机 GPU、还是云端容器（决定 MinerU 推理接入方式）。

2. 生产硬件预算：GPU 型号/显存/并发目标 QPS（决定解析路由阈值与并发策略）。

3. 当前线上最关心的 20 个高价值问题样本（用于阶段0基线与验收）。

4. 是否已有标准引用格式要求（如 GB/T 7714、APA），用于引用一致性校验输出。

5. 是否优先支持中文、英文或双语（影响 embedding/rerank 参数与分词策略）。

6. GraphRAG 是否纳入本期 MVP（若预算紧，可明确延后到下一迭代）。

---

本方案的核心取舍：  

先用 MinerU + Parent-Child + Runtime 治理把“输入质量、检索上下文完整性、执行稳定性”三件事做扎实，再推进 GraphRAG。这样能在可控周期内显著提升输出质量，并保持系统可持续演进。
